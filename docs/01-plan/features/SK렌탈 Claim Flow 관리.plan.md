# SK렌탈 Claim Flow 관리 Plan

> 작성일: 2026-06-14
> 프로젝트 레벨: Starter
> 구현 방식: 단일 정적 HTML 데모 (`index.html`)
> 현재 상태: 1단계 `접수 확인` 상세 팝업 구현 완료, 2~5단계 상세 팝업 구현 예정

## 1. 목표

SK렌탈 Claim 담당자가 Claim 접수부터 지급/정산까지의 업무 흐름을 한 화면에서 확인하고, 각 단계별 빠른 액션 팝업을 통해 필요한 상세정보 확인, 수정, 고객 안내, 처리완료까지 진행할 수 있게 한다.

이번 PDCA 갱신의 핵심 목표는 현재 구현된 `1. 접수 확인` 탭의 실제 동작을 문서화하고, 이후 Claude Code로 구현할 `2. 입고 / 선견적`부터 `5. 지급 / 정산`까지의 설계 기준을 명확히 남기는 것이다.

## 2. 범위

### In Scope

- 현재 구현된 `접수 확인` 단계 빠른 액션 팝업 문서화
- 접수 확인 액션 순서 정의
- 계약정보, 사고정보, 고객 정보 요청, 상담 메모 화면의 데이터/UX 기준 정리
- 2~5단계 빠른 액션 팝업 확장 계획 정의
- 단일 `index.html` 유지
- 외부 프레임워크, 서버, DB, 패키지 추가 없음

### Out of Scope

- 실제 SK전산/API 연동
- 실제 알림톡/문자 발송
- 실제 보험사/AOS/ERP 연동
- 로그인, 권한, 백엔드 저장소
- 디자인 시스템 전면 개편

## 3. 현재 완료 범위: 1. 접수 확인

`접수 확인` 단계의 빠른 액션 흐름은 다음 순서로 구현되어 있다.

1. 계약정보 확인
2. 사고정보 확인
3. 고객 정보 요청
4. 상담 메모 작성 및 접수 확인 완료

각 팝업은 `처리완료` 버튼으로 다음 팝업으로 이동하고, `이전` 버튼으로 이전 팝업으로 돌아갈 수 있다. 첫 단계에서는 `이전` 버튼이 비활성화된다.

## 4. 다음 구현 범위

Claude Code 후속 작업은 아래 4개 Flow 단계의 빠른 액션 팝업을 구성한다.

1. `입고 / 선견적`
2. `수리 승인`
3. `손해사정`
4. `지급 / 정산`

각 단계는 현재 `접수 확인`에서 만든 `ACTION_WORKFLOWS` 패턴을 확장해서 구현한다.

## 5. 성공 기준

- `접수 확인` 문서가 현재 `index.html` 구현과 일치한다.
- 후속 작업자가 문서만 보고 2~5단계 팝업 구현 방향을 이해할 수 있다.
- `CLM-2026-0004`의 `입고 / 선견적`, `업체 미회신`, `AOS 선견적 수신 대기`, `공업사 요청` 테스트 맥락은 유지된다.
- `index.html` 단일 파일 구조를 유지한다.
- 모바일/좁은 화면에서 팝업 내용이 겹치지 않아야 한다.

## 6. 리스크

| 리스크 | 대응 |
|---|---|
| 각 단계 팝업이 커지면서 단일 HTML이 길어짐 | `ACTION_WORKFLOWS` 데이터와 렌더링 함수를 단계별 플래그로 분리 |
| 후속 단계 구현 중 접수 확인 흐름이 깨질 수 있음 | 접수 확인 대표 시나리오를 회귀 체크 항목으로 유지 |
| 브라우저 자동 검증 환경 부족 | 최소한 JS 문법 검사와 수동 클릭 시나리오를 문서화 |
