## 2. 구현 기준

현재 프로젝트는 Starter 레벨의 정적 웹 데모다. 따라서 별도 서버, API, 데이터베이스, 패키지 설치 없이 루트의 `index.html` 하나에서 HTML, CSS, JavaScript를 함께 관리한다.

핵심 화면 구조는 다음 설계를 따른다.

1. 상단 업무 Flow 카드
2. 상단 보조 성과 지표
3. 좌측 조치유형 탭
4. 중앙 Claim 리스트
5. 우측 빠른 처리 패널
6. 액션 결과 토스트

### 3.1 데이터 레이어

## 4. 이번 Do 단계 보정 사항

설계 문서와 현재 구현을 비교한 뒤 작은 차이를 보정했다.

| 변경 전 | 변경 후 | 이유 |
|---------|---------|------|
| `TYPE_ORDER`에 설계 외 `계약 조사` 포함 | 설계 기준 6개 조치유형만 유지 | 설계 문서의 조치유형 탭 기준과 일치 |
| 일부 Claim의 `actionType`이 `계약 조사` | `고객 연락요청`으로 정리 | 화면 탭이 6개 기준으로 렌더링되도록 조정 |
| 컨테이너 최대 폭 `1680px` | `1440px` | 설계 문서의 데스크톱 기준과 일치 |
| 공통 카드 반경 `14px` | `8px` | 설계 문서의 카드 radius 기준과 일치 |

## 5. 수동 테스트 기준

다음 시나리오를 Check 단계에서 우선 검증한다.

1. `index.html`을 브라우저에서 연다.
2. 상단 Flow 카드 중 `입고 / 선견적`을 클릭한다.
3. 중앙 리스트가 `입고 / 선견적` 단계로 필터링되는지 확인한다.
4. 좌측 탭에서 `업체 미회신`을 클릭한다.
5. `CLM-2026-0004` 행을 선택한다.
6. 우측 패널에서 `AOS 선견적 수신 대기`와 권장 액션 맥락을 확인한다.
7. `공업사 요청` 버튼 클릭 시 토스트를 확인한다.
8. `처리완료` 클릭 시 처리상태가 `완료`로 바뀌는지 확인한다.
9. `전체보기` 클릭 시 전체 데이터가 다시 표시되는지 확인한다.

추가로 좁은 화면에서 본문 레이아웃과 테이블 가로 스크롤이 깨지지 않는지 확인한다.

## 6. Check 단계 준비

현재 구현은 설계 문서의 주요 완료 기준을 충족하도록 정리되어 있다.

- 5개 Flow 카드 렌더링
- 6개 조치유형 탭 렌더링
- 12건 이상의 Claim 더미 데이터 표시
- 테스트 고정 데이터 `CLM-2026-0004` 유지
- 처리완료/보류 상태 변경 반영
- 액션 토스트 표시
- B2B SaaS 운영 도구 톤 유지

다음 단계에서는 `$pdca analyze SK렌탈 Claim Flow 관리`로 설계 대비 구현 일치율을 점검한다.
