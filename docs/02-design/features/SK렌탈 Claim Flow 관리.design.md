# SK렌탈 Claim Flow 관리 Design

> 기준 구현 파일: `index.html`
> 현재 설계 반영일: 2026-06-14
> 현재 핵심 완료 범위: `1. 접수 확인` 빠른 액션 팝업

## 1. 화면 구조

기본 화면은 단일 정적 HTML 데모이며, 아래 영역으로 구성한다.

- 상단 업무 Flow 카드 5개
- KPI 라인
- 좌측 조치유형 탭
- 중앙 Claim 리스트
- 우측 빠른 처리 패널
- 빠른 액션 모달
- 토스트 메시지

주요 파일은 `index.html` 하나다. HTML, CSS, JavaScript를 같은 파일 안에서 관리한다.

## 2. Flow 단계

```js
const STAGES = ["접수 확인", "입고 / 선견적", "수리 승인", "손해사정", "지급 / 정산"];
```

현재 상세 팝업까지 설계/구현 완료된 단계는 `접수 확인`이다. 나머지 단계는 후속 구현 대상이다.

## 3. 접수 확인 빠른 액션 설계

### 3.1 액션 순서

`접수 확인` 단계의 빠른 액션 버튼과 팝업 순서는 다음과 같다.

1. 계약정보 확인
2. 사고정보 확인
3. 고객 정보 요청
4. 상담 메모 작성

각 단계는 모달 하단의 `처리완료` 또는 단계별 완료 버튼을 누르면 다음 단계로 이동한다. `이전` 버튼은 현재 팝업의 직전 단계로 이동한다.

### 3.2 계약정보 확인

목적: SK렌터카와 임차인 간 임대차 계약 내용을 확인한다.

표시 방식:

- `상세 설명` 박스는 표시하지 않는다.
- `확인 정보` 영역을 넓게 표시한다.
- 정보는 읽기 전용이다.

주요 항목:

- 렌트/리스 구분
- 상품구분
- 계약번호
- 관리지점
- 최초등록일
- 차량번호
- 모델명
- 잔존가액
- 고객명
- 임차인 구분
- 사고대차 포함
- 정비대차 포함
- 자기부담금
- 현재 조치유형

자기부담금은 Claim 테스트데이터의 `deductible` 값을 사용한다. 값은 아래 셋 중 하나다.

- `손해액의 20% / 20~50만원`
- `30만원`
- `50만원`

### 3.3 사고정보 확인

목적: 담당자가 사고조사에 필요한 정보를 확인하고 필요 시 수정한다.

표시 방식:

- `처리 상태` 박스는 표시하지 않는다.
- `완료 전 체크`를 상단에 표시한다.
- `확인 정보`는 읽기 전용이 아니라 입력 가능한 사고조사 폼으로 표시한다.
- 사고내용은 textarea로 수정 가능해야 한다.

입력 항목:

- 경합 사고접수번호: 보험사 / 사고접수번호
- 경합 사고 담당자: 예시 `김나비 010-****-2481`
- 사고일시
- 사고유형: `차대차 / 추돌`, `차대차 / 차선변경`, `차대차 / 후진사고`, `차대차 / 주/정차 사고`
- 사고장소
- 입고 공업사 및 연락처
- 통보자
- 운전자 및 피보험자와의 관계
- 면허번호
- 경찰서 신고 여부: `신고`, `미신고`
- 견인 여부: `견인`, `미견인`
- 사고내용

입력값은 `accidentEdits`에 Claim별 임시 상태로 유지한다.

### 3.4 고객 정보 요청

목적: 사고접수와 관련해 고객에게 필요한 정보를 알림톡 또는 문자로 요청한다.

표시 방식:

- 사고접수 알림톡 발송 현황을 표로 표시한다.
- 발송 필요 또는 미발송 항목은 표에서 바로 발송할 수 있다.
- 휴대폰 모양 UI에서 알림톡/문자 채널을 선택하고 메시지를 직접 작성해 발송할 수 있다.
- 자주 쓰는 템플릿을 적용하거나 저장할 수 있다.

발송 현황 항목:

- 사고접수 안내
- 운전자 정보 요청
- 사고경위 입력 요청
- 현장 사진 업로드 요청

상태값:

- 발송완료
- 발송필요
- 미발송

템플릿 저장은 `localStorage`의 `claimMessageTemplates` 키를 사용한다.

### 3.5 상담 메모 작성

목적: 고객 상담 내용과 다음 조치를 기록하고 접수 확인 단계를 마감한다.

표시 방식:

- 상담 분류 선택
- 다음 연락 예정 입력
- 상담 메모 textarea 입력
- 자주 쓰는 상담 메모 템플릿 적용
- 기존 상담 이력 표시

완료 버튼 라벨은 `메모 저장 후 접수 확인 완료`이다. 완료 시 상담 메모를 이력에 적재하고 Claim 처리상태를 `완료`로 바꾼다.

## 4. 데이터 모델 추가사항

### 4.1 Claim 테스트데이터

접수 확인 단계 Claim에는 `deductible` 필드가 추가되어 있다.

```js
deductible: "손해액의 20% / 20~50만원"
deductible: "30만원"
deductible: "50만원"
```

### 4.2 접수 확인 워크플로우 상태

```js
const actionProgress = {};
const noticeSends = {};
const accidentEdits = {};
const consultMemos = {};
const consultLog = {};
```

각 상태는 실제 서버 저장이 아니라 정적 데모용 임시 상태다.

## 5. 후속 단계 설계 방향

후속 작업은 `ACTION_WORKFLOWS`에 아래 단계별 워크플로우를 추가한다.

### 5.1 입고 / 선견적

예상 액션:

1. 입고지 확인
2. 선견적 확인
3. 사진요청 발송
4. 공업사 요청

필수 유지사항:

- `CLM-2026-0004`는 `입고 / 선견적`, `업체 미회신`, `AOS 선견적 수신 대기` 맥락을 유지한다.
- 공업사 요청 팝업은 업체 연락/요청 상태를 확인할 수 있어야 한다.

### 5.2 수리 승인

예상 액션:

1. 수리 승인
2. 공업사 오더
3. 고객 안내발송
4. 관리자 확인요청

### 5.3 손해사정

예상 액션:

1. AOS 청구 확인
2. 손해사정 등록
3. 면부책 확인
4. 관리자 검토요청

### 5.4 지급 / 정산

예상 액션:

1. 손해액 확정
2. 공업사 안내
3. SK전산 입력
4. 월정산 반영

## 6. 테스트 시나리오

### 접수 확인 회귀 테스트

1. `접수 확인` Flow 선택
2. 첫 Claim 선택
3. `계약정보 확인` 클릭
4. 계약정보 팝업에서 자기부담금이 단일 필드로 보이는지 확인
5. `처리완료` 클릭 시 `사고정보 확인`으로 이동하는지 확인
6. 사고정보 입력폼이 표시되고 `처리 상태` 박스가 없는지 확인
7. `고객 정보 요청`으로 이동해 알림톡 현황 표와 휴대폰 UI가 겹치지 않는지 확인
8. 발송필요 항목 발송, 직접 메시지 발송, 템플릿 저장 확인
9. 상담 메모 작성 후 접수 확인 완료

### 후속 단계 대표 테스트

1. `입고 / 선견적` Flow 선택
2. `업체 미회신` 탭 선택
3. `CLM-2026-0004` 선택
4. `공업사 요청` 팝업 구현 후 요청/완료 흐름 확인

## 7. 구현 제약

- `index.html` 단일 파일 유지
- 외부 라이브러리 추가 금지
- 서버/API/DB 추가 금지
- 현재 B2B SaaS 운영 도구 톤 유지
- 모바일에서 텍스트/버튼/팝업이 겹치지 않게 유지
