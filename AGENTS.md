# Project Instructions

이 저장소에서 작업하는 모든 에이전트는 아래 규칙을 따른다.

- 항상 한국어로 답변한다.
- 구현 또는 수정 전에 루트의 `todo.md`가 있으면 먼저 읽는다.
- 구현 또는 수정 후 루트의 `spec.md`가 있으면 변경 내용을 반영한다.
- 가능하면 문서 기반 흐름으로 작업한다. 기존 문서와 구현이 다르면 먼저 차이를 확인하고, 필요한 최소 범위만 수정한다.
- 사용자가 명시적으로 요청하지 않은 대규모 리팩터링, 디자인 방향 변경, 기술 스택 변경은 하지 않는다.

# Project Context

- 프로젝트명: `Claim ERP`
- 현재 핵심 기능: `SK렌탈 Claim Flow 관리`
- 프로젝트 레벨: `Starter`
- 현재 PDCA 상태: `do` 단계
- 구현 형태: 외부 프레임워크 없는 단일 정적 웹 데모
- 주요 구현 파일: `index.html`
- 주요 기획 문서: `plan.md`
- 주요 설계 문서: `docs/02-design/features/SK렌탈 Claim Flow 관리.design.md`
- 보조 디자인 자료: `DESIGN.md`
- PDCA 상태 파일: `docs/.pdca-status.json`

이 프로젝트는 Claim 접수부터 지급/정산까지의 처리 흐름을 한 화면에서 보여주는 B2B SaaS형 ERP 데모다. 주요 사용자는 SK렌탈 Claim 담당자, 운영 관리자, 신사업 영업직원, 고객사 임원이다. 화면은 실제 업무 흐름을 유지하되 복잡한 보험사 업무시스템처럼 보이지 않도록 명확하고 정돈된 운영 도구 톤을 유지한다.

# bkit Workflow

세션 시작 시 다음 순서로 프로젝트 상태를 확인한다.

1. `bkit_init`을 호출해 프로젝트 레벨과 PDCA 상태를 초기화한다.
2. 필요하면 `bkit_detect_level`로 레벨을 재확인한다.
3. `bkit_get_status`로 현재 기능과 단계 권장사항을 확인한다. 세션 초기화 오류가 나면 `bkit_init`을 먼저 호출한다.
4. 파일을 쓰기 전에는 가능한 경우 `bkit_pre_write_check`를 호출한다.

현재 프로젝트는 Starter 레벨이므로 기본 파이프라인은 다음 단계를 따른다.

- Phase 1: 기획
- Phase 2: 설계
- Phase 3: 화면/와이어프레임
- Phase 6: UI 구현
- Phase 9: 배포 또는 전달

API, 데이터베이스, 인증, 서버 배포가 명시적으로 요구되지 않으면 Dynamic 또는 Enterprise 수준의 백엔드 설계를 추가하지 않는다.

# Documentation Rules

작업 전 읽기 우선순위:

1. `todo.md`가 있으면 최우선으로 읽는다.
2. `docs/.pdca-status.json` 또는 `bkit_get_status`로 현재 기능과 단계를 확인한다.
3. `docs/02-design/features/SK렌탈 Claim Flow 관리.design.md`를 기준 설계로 삼는다.
4. 구현 세부가 불명확하면 `plan.md`를 확인한다.
5. 디자인 방향과 시각 톤은 `DESIGN.md`를 참고하되, 현재 `index.html`의 B2B SaaS 톤을 우선한다.

작업 후 문서 갱신:

- `spec.md`가 있으면 실제 변경 사항을 반영한다.
- `spec.md`가 없으면 새로 만들지 않는다. 사용자가 요청하거나 기능 명세가 새로 필요할 때만 생성한다.
- 사용자 요청이 문서 정리 자체라면 해당 문서만 수정하고 구현 파일은 건드리지 않는다.

# Implementation Rules

`index.html`은 현재 단일 파일 앱이다. 수정 시 아래 원칙을 지킨다.

- HTML, CSS, JavaScript를 같은 파일 안에서 관리한다.
- 외부 라이브러리, 빌드 도구, 패키지 매니저를 추가하지 않는다.
- 기존 데이터 구조, 렌더링 함수, 필터링 흐름을 유지하면서 필요한 부분만 수정한다.
- 주요 상태는 현재 화면의 Flow 단계, 조치유형 필터, 선택 Claim, 처리상태 변경 흐름과 일관되게 다룬다.
- 더미 데이터는 5개 Flow 단계가 모두 드러나도록 유지한다.
- `CLM-2026-0004`는 테스트 시나리오에 쓰이므로 `입고 / 선견적` 단계와 `업체 미회신` 조치유형 맥락을 깨지 않는다.
- 화면 문구는 한국어 업무 용어를 사용한다. Claim, AOS, ERP처럼 이미 쓰이는 도메인 약어는 유지할 수 있다.
- 운영 담당자가 반복적으로 보는 화면이므로 과한 마케팅 문구, 장식용 그래픽, 랜딩 페이지식 구성은 피한다.

# UI/UX Rules

- 첫 화면은 실제 업무 화면이어야 한다. 랜딩 페이지나 소개 페이지로 바꾸지 않는다.
- 전체 톤은 조용하고 실무적인 B2B SaaS 운영 도구에 맞춘다.
- 핵심 구조는 상단 Flow 카드, 보조 KPI, 좌측 조치유형 탭, 중앙 Claim 리스트, 우측 빠른 처리 패널이다.
- 선택된 Flow 카드, 탭, 테이블 행은 명확히 구분되어야 한다.
- 버튼과 뱃지는 상태 의미가 바로 보이도록 색과 텍스트를 함께 사용한다.
- 모바일/좁은 화면에서도 텍스트가 겹치거나 버튼 안에서 잘리지 않도록 확인한다.
- 카드 안에 불필요하게 카드를 중첩하지 않는다.
- 보라색/남색 계열 단일 팔레트나 장식용 그라데이션에 치우치지 않는다. 현재 블루 기반 ERP 톤은 유지하되 상태색을 균형 있게 사용한다.

# Verification

수정 후 가능한 범위에서 아래를 확인한다.

- `index.html`을 브라우저에서 열었을 때 콘솔 오류가 없는지 확인한다.
- 상단 Flow 카드 클릭 시 해당 단계로 리스트가 필터링되는지 확인한다.
- 좌측 조치유형 탭 클릭 시 현재 Flow 범위 안에서 필터링되는지 확인한다.
- 전체보기 버튼이 전체 데이터 표시로 되돌리는지 확인한다.
- 중앙 Claim 행 클릭 시 우측 패널이 갱신되는지 확인한다.
- 빠른 액션 버튼 클릭 시 토스트 메시지가 표시되는지 확인한다.
- 처리완료/보류 버튼이 선택 Claim의 처리상태를 바꾸는지 확인한다.
- 대표 테스트 시나리오: `입고 / 선견적` -> `업체 미회신` -> `CLM-2026-0004` 선택 -> `공업사 요청` -> `처리완료` -> `전체보기`.

브라우저 검증이 불가능하면 어떤 검증을 못 했는지 최종 답변에 명확히 적는다.

# Response Format

현재 Starter 레벨 응답은 항상 마지막에 아래 항목을 포함한다.

- **Learning Points**: 사용자가 배울 수 있는 핵심 개념 3~5개
- **Next Learning Step**: 다음에 공부하거나 연습할 내용
- 쉬운 용어를 사용하고, 필요하면 "Did you know?" 형식의 짧은 참고를 덧붙인다.

작업 완료 보고는 간결하게 작성한다. 변경 파일, 핵심 변경 내용, 확인한 검증, 남은 리스크를 우선 전달한다.
