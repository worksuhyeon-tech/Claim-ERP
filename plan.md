# Claim ERP Pro — AI-OCR 부품청구서·지급결의·결재 구현 계획

> 대상 저장소: `worksuhyeon-tech/Claim-ERP`  
> 대상 화면: `claim-erp-split/smart-intake.html`의 **청구 견적 정보** 탭  
> 문서 목적: 구현 전 화면·데이터·업무규칙·상태제어·테스트 기준을 확정한다.  
> 현재 단계: Plan only. 이 문서 작성 단계에서는 화면 코드를 수정하지 않는다.

## 1. 목표

Smart접수지의 **청구 견적 정보** 탭에 Pro 기능인 `AI-OCR`을 추가한다.

담당자가 부품청구서를 업로드하면 OCR 결과를 테이블로 전산화하고, 잘못 인식된 부품명·부품번호·부품항목·수량·불인·청구금액·손해사정금액을 직접 수정한 뒤 **개별 지급결의**로 확정한다.

여러 개의 지급결의를 선택하거나 전체선택하여 합산한 뒤 **지급결재 1건**을 상신한다.

```text
결의 001 : 정비청구서  1,500,000원
결의 002 : 부품청구서    500,000원
                         ↓ 전체선택 후 결재상신
결재 001 : 결의 001 + 결의 002
결재금액 : 2,000,000원
```

핵심 원칙:

1. 청구서 1개 = 지급결의 1건
2. 동일 사고건에 지급결의 여러 건 저장 가능
3. 지급결의 여러 건 = 지급결재 1건으로 묶을 수 있음
4. **결의순번과 결재순번은 서로 다른 번호**
5. 결재 LIST에는 개별 결의순번이 아니라 실제 상신한 **결재순번**만 표시
6. OCR 확정과 결재상신은 분리

## 2. 용어와 번호 체계

| 구분 | 정의 | 예시 |
|---|---|---|
| 지급결의 | 청구서 단위로 검토·확정한 지급예정 내역 | 정비청구서 결의, 부품청구서 결의 |
| 결의순번 | 동일 사고건에서 지급결의가 저장된 순서 | 001, 002, 003 |
| 지급결재 | 선택한 지급결의를 합산해 상신하는 결재 건 | 결의 001+002를 결재 001로 상신 |
| 결재순번 | 동일 사고건에서 실제 결재를 상신한 순서 | 001, 002, 003 |
| OCR 원본값 | AI-OCR이 최초 인식한 값 | 수량 2, 부품번호 86511 |
| 담당자 확정값 | 담당자가 검토·수정해 최종 저장한 값 | 수량 1, 부품번호 86511-L1000 |

### 번호 생성 규칙

#### 결의순번

- 사고번호 기준으로 1씩 증가
- 청구유형과 무관하게 저장 순서대로 `001 → 002 → 003`
- 삭제된 번호는 재사용하지 않음

#### 결재순번

- 사고번호 기준으로 실제 결재상신 시 1씩 증가
- 결의 2건을 한 번에 상신해도 결재순번은 1개만 생성
- 반려 또는 상신취소 후 재상신하면 신규 결재순번 생성
- 이전 결재순번과 이력은 유지

## 3. 이번 구현 범위

### 포함

1. 선견적·청구서 전환 영역 우측에 `AI-OCR Pro` 버튼 추가
2. 부품청구서 PDF/JPG/PNG 업로드
3. OCR 분석중 화면
4. 편집 가능한 OCR 결과 테이블
5. 행 추가·행 삭제·불인 체크

7. OCR 원본값과 담당자 수정값 비교 이력 저장
8. 부품 지급결의 확정·수정·선택삭제
9. 지급결의 개별선택·전체선택
10. 선택한 지급결의 합산 지급결재

13. 과실률 미확정 및 변경 제어
14. 결재 LIST에 결재순번·합산금액·포함 결의 표시

- 정비청구서 OCR
- 유리·실런트·렌트 청구서 OCR
- 모델 재학습 파이프라인

## 4. 사용자 흐름

### 4.1 OCR 지급결의 생성

```text
청구 견적 정보 탭 진입
→ 선견적 또는 청구서 확인
→ AI-OCR 클릭
→ 부품청구서 업로드
→ OCR 분석
→ 결과 테이블 확인
→ 담당자가 항목 수정·추가·삭제·불인 처리
→ 금액 및 과실상계 확인
→ 확정
→ 부품 지급결의 신규 저장
```

### 4.2 지급결재 상신

```text
지급결의 저장내역 확인
→ 결재할 결의 개별선택 또는 전체선택
→ 선택 결의 건수·결재금액 확인
→ 다음결재자·결재의견·비밀번호 입력
→ 지급결재 상신
→ 결재순번 신규 생성
→ 선택한 결의는 상신중으로 잠금
→ 결재 LIST에 지급결재 1건 표시
```

## 5. 화면 설계

## 5.1 AI-OCR 버튼

청구 견적 정보의 선견적·청구서 전환 영역 우측에 배치한다.

```text
견적서 전환  [선견적] [청구서]                         [AI-OCR Pro]
```

버튼은 과실률 확정 여부와 관계없이 활성화한다.

- 과실률 미확정: OCR 업로드·분석·결과 수정 가능
- 과실률 확정: OCR 업로드·분석·결과 수정·결의확정 가능
- 기존 지급결재 상신중 또는 완료: 추가 청구 가능성을 고려해 신규 OCR 등록 가능

## 5.2 파일 업로드 모달

제목: `AI-OCR 부품청구서 등록`

표시 정보:

- 사고번호
- 차량번호
- 정비업체
- 드래그앤드롭 영역
- 파일선택
- 선택 파일명·용량
- 분석 시작
- 취소

허용 파일:

- PDF
- JPG/JPEG
- PNG
- 프로토타입 최대 1개, 안내상 최대 20MB

검증 문구:

```text
부품청구서 파일을 선택하세요.
PDF, JPG, PNG 파일만 등록할 수 있습니다.
파일 용량은 20MB 이하만 등록할 수 있습니다.
```

## 5.3 OCR 분석중

```text
부품청구서를 분석하고 있습니다.
파일 구조와 부품 항목을 확인하는 중입니다.
```

프로토타입에서는 짧은 지연 후 더미 OCR 결과로 전환한다.

## 5.4 OCR 결과 테이블

컬럼 순서와 편집 정책:

| 컬럼 | 방식 | 수정 가능 |
|---|---|:---:|
| 순번 | 자동 번호 | 아니오 |
| 부품명 | 텍스트 | 예 |
| 부품번호 | 텍스트 | 예 |
| 부품항목 | 셀렉트/텍스트 | 예 |
| 수량 | 숫자 | 예 |
| 불인 | 체크박스 | 예 |
| 청구금액 | 금액 | 예 |
| 손해사정금액 | 금액 | 예 |

행 기능:

- 행 추가
- 행별 삭제
- 순번 자동 재정렬
- 수량 0 이하 차단
- 음수금액 차단
- 불인 체크 시 손해사정금액 0원
- 불인 해제 시 자동 복원하지 않고 담당자가 손해사정금액 직접 입력

## 5.5 금액 요약

```text
청구금액 합계      = 각 행 청구금액 합계
손해사정금액 합계  = 각 행 손해사정금액 합계
과실상계금액       = 손해사정금액 합계 × 자차 과실률
최종 지급예정금액  = 손해사정금액 합계 - 과실상계금액
```

과실상계는 **손해사정금액에만 적용**한다. 청구금액에는 적용하지 않는다.

표시 예시:

```text
                         청구금액             손해사정금액
과실상계금액                    -                  84,000원
합계금액                  500,000원               336,000원
```

프로토타입은 원 단위 정수 계산을 사용한다. 실제 절사·반올림 정책은 추후 보험금 산정 IF 명세에 맞춰 교체한다.

## 6. 과실률 제어

## 6.1 과실률 미확정

청구 견적 정보 탭과 OCR 화면 진입 자체를 차단하지 않는다.

허용:

- 탭 진입
- 선견적·청구서 조회
- AI-OCR 업로드
- OCR 분석
- OCR 결과 수정

비활성:

- OCR 결과 `확정`
- 기존 지급결의 `재확정`
- 지급결의 선택 체크박스·전체선택
- 지급결의 선택삭제
- 하단 지급결재 상신 영역

제어 문구는 정확히 아래와 같이 표시한다.

```text
과실율 확정 후 결의입력하세요
```

표시 위치:

1. OCR 결과의 확정 버튼 주변
2. 하단 지급결의·결재 영역 상단 또는 비활성 오버레이

## 6.2 지급결의 저장 후 과실률 변경

미결재 지급결의:

- 상태를 `재확인 필요`로 변경
- 기존 과실률과 현재 과실률 표시
- 최신 과실률로 과실상계 자동 재계산
- 담당자가 `재확정`해야 결재 선택 가능

상신중 지급결의:

- 진행 중 결재금액을 자동 변경하지 않음
- 경고 표시
- 실제 운영 시 자동회수 여부는 별도 정책으로 확정

결재완료 지급결의:

- 기존 금액 변경 금지
- 추가지급·환입은 신규 결의로 처리

## 7. 하단 지급결의·결재 레이아웃

기존 결재 컴포넌트를 다음 구조로 변경한다.

```text
┌──────────────────────────────┬──────────────────────────────┐
│ 지급결의 저장내역             │ 결재순번 / 결재이력            │
│                              │                              │
│ ☐ 전체선택                    │ 결재 001  결재완료              │
│ ☐ 001 정비청구서 ...          │ 포함결의 001, 002              │
│ ☐ 002 부품청구서 ...          │ 결재금액 2,000,000원           │
│                              │                              │
│ [수정] [선택삭제]             │ 결재 002  반려                  │
└──────────────────────────────┴──────────────────────────────┘
┌─────────────────────────────────────────────────────────────┐
│ 선택 결의 2건 / 결재금액 2,000,000원                         │
│ 다음결재자 / 결재의견 / 비밀번호                              │
│                                      [상신취소] [지급결재 상신] │
└─────────────────────────────────────────────────────────────┘
```

## 7.1 왼쪽: 지급결의 저장내역

컬럼:

- 선택
- 결의순번
- 결의종류
- 원본 파일명/입력출처
- 청구금액
- 과실상계 전 손해사정금액
- 과실상계금액
- 최종 지급예정금액
- 상태
- 저장일시
- 입력자

기능:

- 개별선택
- 전체선택
- 상세보기
- 수정
- 선택삭제

전체선택은 현재 선택 가능한 결의만 체크한다. 상신중·결재완료 결의는 포함하지 않는다.

## 7.2 오른쪽: 결재순번

표시 정보:

- 결재순번
- 포함 결의순번
- 결재금액
- 결재상태
- 상신일시
- 결재완료일시
- 상신자
- 결재자
- 결재의견

결재순번은 지급결의를 저장할 때가 아니라 **실제 결재를 상신할 때만** 생성한다.

## 7.3 하단: 결재상신

- 선택 결의 건수
- 선택 결의 최종 지급예정금액 합계
- 다음결재자
- 결재의견
- 비밀번호
- 상신취소
- 지급결재 상신

선택 결의가 없거나 과실률이 미확정이면 지급결재 상신 버튼을 비활성화한다.

## 9. OCR 수정이력

키 입력마다 저장하지 않는다.

`확정` 또는 `재확정` 시 OCR 원본과 최종값을 비교하여 변경된 필드만 저장한다.

```javascript
{
  rowSeq: 1,
  field: "partNumber",
  fieldLabel: "부품번호",
  before: "86511",
  after: "86511-L1000",
  changedBy: "EMP004",
  changedAt: "2026-07-15 14:30"
}
```

추가로 기록할 변경 유형:

- OCR 행 수정
- OCR 오인식 행 삭제
- OCR 누락 행 직접 추가
- 불인 변경

향후 분석지표:

- OCR 무수정 확정률
- 부품명 수정률
- 부품번호 수정률
- 수량 수정률
- 청구금액 수정률
- 손해사정금액 수정률
- 불인 변경률
- 추가·삭제 행 수
- 업체/양식별 수정률

## 10.2 지급결재

기존 `APPROVALS`에 다음 필드를 확장한다.

```javascript
{
  id: "APR-001",
  claimNo: "CLM-2026-0007",
  approvalSeq: "001",
  approvalType: "지급(종결)",
  resolutionIds: ["RES-0001", "RES-0002"],
  includedResolutionSeqs: ["001", "002"],
  approvalAmount: 1760000,
  approvalStatus: "상신중"
}
```

호환 원칙:

- 신규 데이터는 `approvalSeq` 사용
- 기존 시드 데이터는 `resolutionNo`를 사용하므로 화면에서는 `approvalSeq || resolutionNo`로 표시
- 지급결의의 `resolutionSeq`와 결재의 `approvalSeq`를 혼용하지 않음

## 11. 결재 연동

### 결재금액

```text
결재금액 = 선택된 지급결의의 최종 지급예정금액 합계
```

### 상신

1. 선택 결의 상태 재검증
2. 과실률 확정 여부 재검증
3. 결의 확정 당시 과실률과 현재 과실률 비교
4. 결재순번 생성
5. `APPROVALS`에 지급결재 1건 저장
6. 선택 결의 상태를 `상신중`으로 변경
7. 각 결의의 `currentApprovalId` 연결
8. 결재 LIST 갱신

### 반려

- 연결 결의 상태 `반려`
- 수정·삭제·재상신 가능
- 이전 결재 ID는 이력 유지

### 상신취소

- 연결 결의 상태 `상신취소`
- 수정·삭제·재상신 가능
- 이전 결재 ID는 이력 유지

### 결재완료

- 연결 결의 상태 `결재완료`
- 수정·삭제·재상신 불가
- 결재금액과 포함 결의 고정

## 16. 필수 테스트

### TC-01 OCR 정상

과실률 확정 사고건에서 PDF 업로드 후 확정한다.

기대: 부품 지급결의 001 저장, 입력방식 AI-OCR 표시

### TC-02 OCR 수정이력

부품번호·수량·금액 수정, 행 추가·삭제 후 확정한다.

기대: 수정필드·추가행·삭제행 이력 저장

### TC-03 과실률 미확정

OCR 업로드·수정 후 확정과 하단 영역을 확인한다.

기대: OCR 검토 가능, 결의확정·결재상신 비활성, 지정 문구 표시

### TC-04 다중 결의

정비 결의 001이 있는 상태에서 부품 OCR을 확정한다.

기대: 부품 결의 002 저장, 001 유지

### TC-05 전체선택 결재

결의 001·002를 전체선택하여 상신한다.

기대: 결재 001 한 건 생성, 포함 결의 001·002, 합산금액 표시

### TC-06 일부선택 결재

결의 001·003만 선택하여 상신한다.

기대: 001·003만 잠금, 002는 결재 전 유지

### TC-07 상신중 잠금

상신중 결의의 수정·삭제를 시도한다.

기대: 선택·수정·삭제 모두 불가

### TC-08 반려

결재 반려 후 연결 결의를 수정·삭제·재상신한다.

기대: 모두 가능, 재상신 시 신규 결재순번

### TC-09 상신취소

상신취소 후 연결 결의를 수정·재상신한다.

기대: 잠금 해제, 신규 결재순번

### TC-10 결재완료

최종결재 후 연결 결의를 수정·삭제한다.

기대: 모두 불가

### TC-11 과실률 변경

미결재 결의 저장 후 과실률을 변경한다.

기대: `재확인 필요`, 결재선택 불가, 재확정 후 가능

### TC-12 회귀

기대:

- 선견적·청구서 전환 정상
- 차량 사진·확대·주석 정상
- 결재 비밀번호 정상
- 결재 LIST 검색·필터·상세 정상

## 17. 완료 조건

1. 담당자가 OCR 결과 전 항목을 수정할 수 있다.
2. 과실률 미확정이어도 OCR 검토는 가능하지만 결의확정은 불가능하다.
3. 청구서별 지급결의가 여러 건 저장된다.
4. 지급결의를 개별선택·전체선택할 수 있다.
5. 선택 가능한 결의만 삭제·결재할 수 있다.
6. 여러 지급결의를 지급결재 1건으로 합산할 수 있다.
7. 결의순번과 결재순번이 분리된다.
8. 결재 LIST에는 실제 상신된 결재순번만 표시된다.
9. 상신중·결재완료는 수정·삭제 불가다.
10. 반려·상신취소는 수정·삭제·재상신 가능하다.
11. 과실률 변경 시 미결재 결의는 재확정 전까지 결재 불가다.
12. OCR 원본과 담당자 수정값 차이가 이력으로 남는다.
13. 기존 Smart접수지·결재 LIST 기능이 깨지지 않는다.

## 18. 업무 기대효과

- 청구서별 결의 분리관리
- 일부 또는 전체 결의 선택결재

- 상태에 따른 수정 가능 여부 즉시 확인

## 19. 구현 금지사항

1. 결의순번과 결재순번을 같은 값으로 사용하지 않는다.
2. OCR 확정과 지급결재 상신을 한 버튼으로 처리하지 않는다.
3. 과실률 미확정 상태에서 지급결의를 저장하지 않는다.
4. 전체선택에 상신중·결재완료 결의를 포함하지 않는다.
5. 결재완료 결의를 수정·삭제하지 않는다.
6. OCR 결과를 담당자 검토 없이 자동 확정하지 않는다.
7. 기존 선견적·청구서·사진뷰어·결재 기능을 제거하지 않는다.
8. 화면별 분리 구조를 단일 HTML 구조로 되돌리지 않는다.
